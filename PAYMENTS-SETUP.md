# Payments & per-voter pricing (Verita)

Verita prices each election by its **organiser-declared registered (enrolled) voter
count**. The total count selects one bracket and every voter is billed at that flat
rate (money spec §6). The first 50 voters are free, so small elections launch with no
payment. Payment is taken **before launch** — an election can only be opened once paid
(or if it's in the free tier).

## Rate card (default)

| Registered voters | ₦ / voter | $ / voter |
|---|---|---|
| 1 – 50 | Free | Free |
| 51 – 500 | ₦120 | $0.25 |
| 501 – 2,500 | ₦90 | $0.18 |
| 2,501 – 10,000 | ₦55 | $0.12 |
| 10,001 – 50,000 | ₦35 | $0.08 |
| 50,000+ | ₦22 (volume) | $0.045 |

Override with the `PRICING_TABLE` env var (JSON) if needed — see `.env.example`.
Brackets live in `app/src/services/pricing.ts`; unit tests in `pricing.test.ts`
verify the spec's worked examples.

## Rails — Monnify primary, Paystack fallback

`PAYMENT_PROVIDER=squad` (default) uses Squad when its secret key is present, and
falls back to Monnify automatically if Squad isn't configured. Set
`PAYMENTS_ENABLED=true` to turn the paywall on. (Paystack and the Lemon Squeezy
subscription have been retired.)

### Squad (GTBank / HabariPay — payment links)

```
SQUAD_SECRET_KEY=sandbox_sk_xxxxxxxx               # live: sk_xxxxxxxx
SQUAD_BASE_URL=https://sandbox-api-d.squadco.com   # live: https://api-d.squadco.com
SQUAD_PAY_BASE_URL=https://sandbox-pay.squadco.com # live: https://pay.squadco.com
# Optional static dashboard product link (manual fallback only):
# SQUAD_PAYMENT_LINK=https://pay.squadco.com/toramavote
```

Flow (`app/src/services/squad.ts`):
1. **Create link** — `POST /payment_link/otp` makes a one-off hosted payment link for
   the **exact metered amount** (in kobo/subunits). The link `hash` is our own
   reference (`vote-<id>-<token>`), so the checkout URL is deterministic:
   `<SQUAD_PAY_BASE_URL>/<reference>`. The organiser is redirected there.
2. **Reference round-trip** — the link's `redirect_link` is our callback with
   `?ref=<reference>`, so when Squad sends the payer back we know which payment it was
   (the payment-link transaction ref is generated by Squad).
3. **Verify (authoritative)** — `GET /transaction/verify/{transactionRef}`; confirms
   the status is `success` and the settled amount ≥ the metered amount before the
   election is marked paid + opened. Used by both the redirect callback and the webhook.

### Dashboard setup (Profile → API & Webhook)

```
Webhook URL:  https://vote.torama.money/webhooks/squad
Redirect URL: https://vote.torama.money/account/pay/callback
```

### Webhook

`POST /webhooks/squad` is verified with the **uppercase-hex HMAC-SHA512** of the raw
body using `SQUAD_SECRET_KEY` in the `x-squad-encrypted-body` header, then re-verified
server-side. The link-payment webhook carries Squad's transaction ref + the payer's
email + amount (not our reference), so it's matched to the most recent pending Squad
payment by email + amount, then verified. Monnify's `/webhooks/monnify` is unchanged.

## Deploying these settings

VOTE's `.env` is **hand-maintained on the server** at `/opt/vote/.env` — the CI deploy
just runs `git pull && docker compose up -d --build` over SSH and does **not** template
`.env` from GitHub Secrets. To enable Squad in production:

```bash
ssh user1@vote.torama.money
cd /opt/vote
nano .env          # add the SQUAD_* lines above with LIVE URLs;
                   # set PAYMENT_PROVIDER=squad and PAYMENTS_ENABLED=true
docker compose up -d --build
curl -s http://127.0.0.1:8090/healthz   # {"status":"ok"}
```

## Notes

- The `paystack_payments` table is provider-agnostic (`provider`,
  `provider_reference`, `voters` columns added by the idempotent boot migration);
  new rows store `provider='squad'`.
- Setting the wrong voter count: organisers edit the registered-voter count on the
  election edit page while the election is still a draft.
